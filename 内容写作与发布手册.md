## 一、写作规范

### 1. 写给谁、写成什么样

| 原则 | 说明 |
|------|------|
| **读者** | 分子模拟 / ML 方向的科研学习者：会一点 Python，不一定熟 Linux、集群、Git |
| **语气** | 通俗、口语化，像师兄带师弟——少堆术语，多讲「这一步解决什么问题」 |
| **结构** | 先场景、再步骤、再表格小结；长文用 `## 一、二、三、` 分节 |
| **篇幅** | 单篇讲清一个主题；细节拆到专题文，总览文只串链路 |

**导语（开篇 / 前言）应包含：**

1. 这篇解决什么痛点（用具体场景，如「轨迹几 GB、Git push 卡死」）  
2. 本文讲什么、**不讲什么**（与姊妹篇分工）  
3. 前置条件（如「先配好 SSH」「假设已有 myenv」）

#### 1.1 导语不能过于简单（尤其 VIP）

导语是读者决定「往下读 / 开通会员」的第一屏，**禁止**写成两三句干巴巴的定义 + 「本文讲 X」。要有吸引力：先抛场景或痛点，再亮出这篇能带走什么，必要时用一小表点名关键概念或应用（如输运系数对上电池、热管理）。

| 要求 | 说明 |
|------|------|
| **厚度** | 一般 **3–6 段**（可含一张小表）；不要只有「X 是什么 + 本文讲 X」两段交差 |
| **钩子** | 开头用具体场景、反差或「你会卡住的问题」，少用「本文系统介绍……」 |
| **地图感** | 让读者立刻感到「这张谱系 / 这条流程和我有关」——点名关键量、适用场合或与姊妹篇分工 |
| **VIP 更严** | `paywall: vip` 时，**付费墙外的导语几乎是整篇的免费门面**：必须写够吸引力与信息密度，让人愿意往下读 / 开通；勿把干货全塞进墙内、墙外只留空壳定义 |
| **仍要克制** | 吸引 ≠ 标题党；不剧透墙内逐步操作的全部细节，但墙外要说清「学完能解决啥、和谁对照」 |

**偏弱（避免）：**

> 输运系数描述体系对扰动的响应。本文介绍扩散、粘度与热导，以及平衡与非平衡算法。

**更好（示意）：**

> 密度、RDF 回答「平衡在哪」；电池电解液「离子跑多快、内阻大概怎样」、冷却液「导热好不好」，问的是另一类量——扩散、电导、粘度、热导。先认清谱系里有谁、对上哪些应用，再选平衡涨落还是非平衡驱动……

VIP 结构仍见本手册「发布与上线 · 付费与 VIP」：导语留在墙外免费；墙外导语按上表写厚，不要为了「省字」把前言砍成定义句。

**篇末建议有：**

- **小结**（3–5 条可执行的要点）  
- **前置阅读 / 下一步**（用文章标题链出，见第 3 节）  
- 可选 **常见问题**（FAQ）

### 2. 通俗化写法

语气要好懂，但**不要用元标注告诉读者「下面要讲人话了」**。直接写清楚即可。

#### 2.1 禁止的元话头（直接删掉，改写成正文）

| 避免（正文 / 标题 / 表头） | 说明 |
|---------------------------|------|
| 「人话」「读成人话」「讲人话」 | 像在嘲讽读者；删掉标签，把解释写进正文 |
| 「通俗地讲」「通俗讲」「大白话」 | 同上 |
| 「白话：」「白话」作表头 / 小节标题 | 表头改用「含义」「说明」「直观理解」等 |
| 「简单说：」「说白了：」当段首标签 | 可保留「也就是说」等自然过渡，不要当「降智开关」 |
| 「认门牌」「先认门牌」 | 像在暗示「只背名字不用懂」；改成「常见名字 / 常见做法 / 先分清种类」等直接说法 |
| 「选型」作标题、表头或套话（「选型总闸」「选型梯子」「选型速查」） | 改成「怎么选」「如何判断用哪条路」「适用场合」；动词可用「选择」「判断」 |

**对：** 直接写「关联长度可以想成……」  
**错：** 「关联长度是啥？（人话）」「白话：关联长度就是……」

**对：** 「常见热浴做法见下表」「按问题判断用经典力场还是 AIMD」  
**错：** 「常见热浴（认门牌）」「力场选型总闸」

#### 2.2 用场景和比喻，不用空话

| 避免 | 改为 |
|------|------|
| 「本文系统介绍 X 的配置与最佳实践」 | 「连 Git、登集群、传轨迹都离不开 SSH；配好一次密钥，以后不用一遍遍敲长域名」 |
| 「版本控制工具」 | 「项目的时光机，避免 `分析_最终版_v3.ipynb`」 |
| 「交互式开发环境」 | 「浏览器里边写边跑的 Notebook 工作台」 |

比喻要贴切、一两句即可，不要通篇比喻。

#### 2.3 表格承担「对照」和「清单」

- 组件对照（工具 → 含义 / 直观理解 → 详细教程）  
- 适合 / 不适合（如 Notebook 适合探索、不适合几天几夜生产跑）  
- 排错速查（现象 → 可能原因 → 怎么办）

#### 2.4 Tips 块

非显而易见、但省时间的点，用引用块标出：

```markdown
> **Tips：** 登录节点能否访问外网装包，各校不同——装之前先问管理员。
```

可选前缀：`Tips`、`可选`、`注意`。不要滥用，一篇 3–6 处为宜。

#### 2.5 命令与代码

- 命令注明**在哪跑**（本机 WSL / 登录节点 / 计算节点）  
- 长命令可拆步，关键参数用行内注释说明  
- 复制即用的片段优先；理论推导留给知识文档（K 系列）

#### 2.6 姊妹篇分工要说清

同一主题拆成「工具操作」与「科研规范」时，开篇对照表写明白，避免读者不知道先读哪篇。

**示例（Jupyter）：**

| 主题 | 读哪篇 |
|------|--------|
| 安装、界面、单元格、快捷键 | [JupyterLab简明教程](在线资源/01-技术文档/T11-JupyterLab简明教程.md) |
| 目录、复现、Git、清输出 | [Jupyter Notebook科研使用规范](在线资源/01-技术文档/T16-Jupyter Notebook科研使用规范.md) |

### 3. 引用与互链（不要编号）

#### 3.1 读者可见文字：禁止 T## / E## / K##

**正文、表格、链接锚文字**里不要出现文章编号，一律用**完整标题**或自然语言。

| 错误 | 正确 |
|------|------|
| 见 [T10](T10-集群与SLURM简明教程.md) | 见 [集群与SLURM简明教程](T10-集群与SLURM简明教程.md) |
| 见 T01 第七节 | 见 [分子模拟工作平台搭建](T01-分子模拟工作平台搭建.md) 第七节 |
| 跟一篇 **C01 绘图** 练手 | 跟一篇 [NumPy与Matplotlib简明教程](C01-NumPy与Matplotlib简明教程.md) 练手 |
| 工作流 T12–T18 | 工作流与写作（目录规范、数据备份、论文图等） |
| K01–K31 概念文 | [知识文档](在线资源/00-知识文档/README.md) 栏目下的概念文 |

**允许出现编号的地方（读者一般看不到或属内部管理）：**

- YAML 字段 `id: T01`  
- **文件名** `T10-集群与SLURM简明教程.md`（链接目标路径）  
- 本手册「内容规划」、编辑用总表  
- [资源导航.md](在线资源/资源导航.md) 等内部导航表中的 `id` 列（维护用，正文仍用标题链）

#### 3.2 链接锚文字 = 文章标题

```markdown
<!-- 好 -->
[SSH密钥与config配置简明教程](T08-SSH密钥与config配置简明教程.md)
[NumPy与Matplotlib简明教程](../02-实战案例/C01-NumPy与Matplotlib简明教程.md)

<!-- 差 -->
[SSH](T08-SSH密钥与config配置简明教程.md)
[远程连接集群](T07-VSCode与Cursor远程连接集群.md)   <!-- 应写完整标题 -->
[T01 第二节](T01-分子模拟工作平台搭建.md)           <!-- 应写标题 + 节名 -->
```

节号可以跟在标题后：**「[分子模拟工作平台搭建](…) 第二节」**，不要用编号代替标题。

#### 3.3 本地路径约定

| 场景 | 写法 |
|------|------|
| 同目录技术文互链 | `[Git简明使用教程](T04-Git简明使用教程.md)` |
| 跨栏目（技术 → 案例） | `[NumPy与Matplotlib简明教程](../02-实战案例/C01-NumPy与Matplotlib简明教程.md)` |
| 技术 → 知识文档 | `[分子动力学模拟概述](../00-知识文档/K02-分子动力学模拟概述.md)` |
| 技术 → 资源导航 | `[资源导航](../资源导航.md)` |

- 源稿里**不要写 WordPress 绝对 URL**；发布时由 `publish.py` 替换。  
- 未发布文章在 Obsidian 里仍可链 `.md`；线上未发布链显示为待发布（见本手册「发布与上线」）。

#### 3.4 尚未成稿的选题

无对应 `.md` 文件时，**不要做空链**。可写纯文字并注明状态：

```markdown
- Docker与Apptainer入门（规划中）—— 环境复现的另一条路
```

#### 3.5 站内脉络 vs 单篇阅读

- **单篇文末**：「前置阅读 / 下一步」用标题链接即可。  
- **全站顺序、编号块**：只放在 本手册「内容规划」，**不要**在公开正文中写「按 T01→T11 顺序读完」。

### 4. 文首元数据与系列标签

#### 4.1 YAML（真源）

```yaml
---
id: T01
title: 分子模拟工作平台搭建
series: 在线资源
tier: 01-技术文档
status: draft
topic: 平台搭建
paywall: free
---
```

知识/技术文 YAML 里 `paywall` 现用：`free` · `vip` · `download-vip` · `download-paid`（旧 `none` 等同 `free`）。详见本手册「发布与上线 · paywall」。

- **`id` / `title`**：`id` 供脚本与内部表；`title` 即对外标题，也是互链时用的名称。  
- **`topic`**：站点分类下的主题词，与 `系列标签` 第三、四格呼应即可。

#### 4.2 系列标签（正文第一行引用块）

紧跟 YAML 后写一行，**固定 4 个**反引号标签，用 ` · ` 分隔：

```markdown
> **系列标签：** `技术文档` · `平台搭建` · `MolSimulX` · `环境配置`
```

| 栏目 | 第 1 格 | 第 2–4 格示例 |
|------|---------|----------------|
| 01 技术 | `技术文档` | 主题词，如 `SSH`、`工作流`、`科研绘图` |
| 00 知识 | `知识文档` | `分子模拟`、`力场`、`MolSimulX` 等（共 4 格） |
| 文件名 | `K02-分子动力学模拟概述.md` | 与 `T##-` / `C##-` 一致；正文互链只用标题 |
| 02 实战 | `实战案例` | 体系/场景名（如 `LJ流体`、`水盒子`）；勿用纯工具名当主标签 |

标签要**能检索、能概括**，不要凑字数；同一主题姊妹篇标签要有区分度（如 `JupyterLab` vs `可复现`）。

### 5. 章节与排版习惯

- **一级结构**：`## 一、……`、`## 二、……`（中文序号 + 顿号）  
- **小节**：`###` 即可，不必再写「1.1」除非表格需要  
- **分隔**：大节之间用 `---`  
- **引用块里不要嵌套围栏代码**：`>` 引用中写 bash 等围栏代码块时，多数 Markdown / WP 渲染会错乱（出现 `"bash`、代码块样式异常）。做法：引用只写说明，**代码块放在引用外**的普通段落里。  
- **加粗与全角括号（Obsidian）**：`**…）**` 紧贴后面的汉字时，预览常把 `**` 显示成字面量（如 `有限元（FEM）**等`）。写法二选一：  
  - 推荐：`**有限元**（FEM）等`（加粗不含全角括号）  
  - 或：`**有限元（FEM）** 等`（闭 `**` 后留空格再接下文）  
  同理避免 `）**字`、`字**（` 粘连导致的边界错误。  
- **图片**：Obsidian 可用 `![[name.png]]`；发布前由 `sync_article_images.py` 归档为 `images/articles/.../web/*.webp`  
- **封面图与正文**：**导语（前言）不要解释封面图**——图放在导语之后、第一节之前即可，下面不要跟「上图是……」「封面画的是……」这类说明。若图与正文某节直接相关，在**正文该节**里自然带一句（如「封面左图……」「与封面上的烧瓶对照……」）；只是装饰、或和文意对不上，**不要硬写**——在协作里标明「封面图不太贴」，请作者更换。  
- **封面显示**：发布时正文**第一张图**会标为封面；封面与正文插图在站点上均为 **`max-height: 420px`**、`object-fit: contain`（比例自由、不裁切、水平居中）。源图像素与比例不必统一；优先横图。  
- **外链**：官方文档、ColorBrewer 等用 `https://` 完整 URL  
- **付费/VIP**：结构见本手册「发布与上线 · 付费与 VIP」，此处不重复

### 6. 审阅自检清单

发布前或 PR 前过一遍：

- [ ] 导语是否说清「写给谁、解决啥、前置啥」；是否**够厚、有场景钩子**（忌两三句空壳定义）  
- [ ] **VIP 篇**：墙外导语是否已有吸引力与信息密度（勿只留定义句等开通）  
- [ ] 语气是否口语化，有没有未解释的首字母缩写  
- [ ] 是否避开「人话 / 通俗地讲 / 认门牌 / 选型」等禁止套话（见 §2.1）  
- [ ] 封面图是否与主题相关；**导语下无「上图/封面」解说**；需要时是否在正文结合封面描述；不相关是否已提示更换  
- [ ] 加粗是否避开 `）**汉字`（应用 `**术语**（英文）` 或闭 `**` 后空格）  
- [ ] 互链是否用**完整标题**；跨目录路径是否正确  
- [ ] 系列标签是否 4 个、与 `topic` 一致  
- [ ] 与姊妹篇分工是否写清（避免 T11/T16 类重复）  
- [ ] 篇末是否有小结 + 前置/下一步  
- [ ] `status` 是否与审阅阶段一致（见本手册「发布与上线」）

#### 可选：批量检查编号残留

在 `在线资源/01-技术文档/` 下对 T01–T18 等跑检查（排除 YAML `id:` 与链接文件名）：

```bash
cd 在线资源/01-技术文档
python3 << 'PY'
import re
from pathlib import Path
pat = re.compile(r'(?<![/\w-])\b[TEK]\d{2}\b(?![\w-])')
for f in sorted(Path('.').glob('T*.md')):
    for i, line in enumerate(f.read_text().splitlines(), 1):
        if line.strip().startswith('id:'): continue
        if pat.search(line) and not ('](' in line and pat.search(line.split('](')[-1] or '')):
            print(f'{f.name}:{i}: {line.strip()[:100]}')
PY
```

有输出则需改写成标题或自然语言。

### 快速发布指南

> 只记命令时用本节；状态、YAML、付费、模板等细节见下文各章。

### 一次性准备

```bash
cd /path/to/MolSimulX网站

python3 -m venv .venv && source .venv/bin/activate   # 或 conda activate molsimulx
pip install -r tools/requirements-publish.txt

cp .env.local.example .env.local
# 编辑 .env.local：WP_SITE_URL、WP_USERNAME、WP_APP_PASSWORD、WP_TEMPLATE_POST_ID=650
```

### 日常流程（单篇）

```bash
# 1. 看编号与 status
python tools/publish.py --list

# 2. 审阅通过后，在 md 文首 YAML 把 status 改为 reviewed（或 revised）

# 3. 预览转换（不上传 WP、不整理图）
python tools/publish.py T01 --dry-run

# 4. 发布到 WP 草稿（自动整理配图 + 注入模板 650；默认 WP 状态 draft）
python tools/publish.py T01 --write-back-id

# 5. 到 WP 后台预览 → 配 erphpdown → 人工点「发布」
# 6. 本地 YAML 改 status: published，填写 wp_post_id、published_at
```

`T01` 可换成任意编号；也可用 `--id T01`、`--pick`（交互选篇）、`--file Git简明使用教程.md`（唯一文件名）。

### 日常流程（批量）

`publish.py` **已支持一次处理多篇**：按顺序逐篇执行配图同步 → 转 HTML → 上传 WP；结束时打印 **批量发布汇总**（成功 / 跳过 / 失败）。

```bash
# 指定多篇（空格分隔）
python tools/publish.py T02 T03 T04 --write-back-id

# 或用 --id（支持逗号、空格：T02,T03 或 T02 T03）
python tools/publish.py --id T02,T03,T04 --write-back-id

# 全库扫描：遍历 在线资源 / 在线工具 / 解决方案 下全部教程 md
# 仅 status=reviewed / revised 的会真正同步，其余在汇总里显示「跳过」
python tools/publish.py --all-reviewed --write-back-id

# 批量预览（不上传 WP）
python tools/publish.py T01 T02 T03 --dry-run
```

**批量时注意：**

| 项 | 说明 |
|----|------|
| **status 门禁** | 每篇独立判断；`draft` / `published` 默认跳过，汇总里可见原因 |
| **共用参数** | `--dry-run`、`--write-back-id`、`--publish`、`--force`、`--skip-sync-images` 等对**整批**生效 |
| **退出码** | 任一篇 HTTP/校验失败则进程退出码为 `1`；成功的篇目不受影响 |
| **发布顺序** | 显式指定 ID 时按你写的顺序；`--all-reviewed` 按目录树路径排序 |
| **站内互链** | 被引用文尚无 `wp_post_id` 时，引用方线上仍为「待发布」；批量首发后需**再发一遍引用它的文章**（见第 5 节） |

建议批量首发：`00` 知识 → `01` 技术 → `02` 示例`，或按本手册「内容规划」学习路径指定 ID 列表，减少站内灰链。

### 2. 文章状态（工作流）

| status | 含义 | 对外 | 发布动作 |
|--------|------|------|----------|
| `draft` | AI 初稿 / 审阅中 | 否 | 不发布 |
| `reviewed` | 首次人工审阅通过 | 否 | **CREATE**（新建 WP 文章） |
| `published` | 已上线，本地与线上一致 | 是 | 无 |
| `revised` | 已上线但本地又修改 | 线上仍为旧版 | **UPDATE**（同 `wp_post_id`） |
| `archived` | 下线或废弃 | 否 | 不发布 / 可选下线 |

#### 状态流转

```text
draft ──审完──► reviewed ──首次发布──► published
  ▲                                      │
  └──────── 大改退回 draft                │ 本地修改
                                          ▼
                                    revised ──再审 + 更新发布──► published
```

- **`reviewed`**：只用于第一次上线。  
- **`revised`**：只用于已发布后的修改；更新成功后改回 `published`。  
- 自动发布工具（若实现）应：**`reviewed` / `revised` → 写入 WP 均为 `draft`**；人工在后台点「发布」后，本地才改 `published`。

#### WordPress 侧：先草稿，再人工发布

```text
本地 status=reviewed（或 revised）
        ↓
脚本 / 手工同步 → WP 文章状态 = draft（草稿）
        ↓
WP 后台预览（模板 650、图片、erphpdown、VIP）
        ↓
人工点击「发布」
        ↓
本地 status=published，填写 wp_post_id、published_at
```

| 阶段 | 本地 YAML | WordPress |
|------|-----------|-----------|
| 写作审阅 | `draft` | （无或旧版） |
| 可上线 | `reviewed` | 待创建 **draft** |
| 已同步待确认 | `reviewed` | **draft**（预览） |
| 正式上线 | `published` | **publish** |
| 改稿 | `revised` | 再次 **draft** 或预览后更新（见下） |

**修订说明：** 已上线的文章若在 WP 里改回 `draft` 会暂时下架。可选做法：（1）严格审核：修订也先 draft，确认后再发布；（2）已熟内容：用 WP「预览修订」插件或仅在本地 `revised` 审完后再点一次「更新」。默认约定：**自动工具只创建/更新 draft，不直接 publish**。

### 3. 文章编号、文件名与文首 YAML

#### 文件名要不要加 `T01-` 前缀？

**建议：新文章可以加，旧文不必一次性重命名。**

| 做法 | 优点 | 缺点 |
|------|------|------|
| **仅 YAML 里写 `id: T01`** | 文件名仍可读；Obsidian 双链不用改 | 资源管理器里不按编号排序 |
| **文件名 `T01-标题.md`** | 文件夹里一眼排序、好追踪 | 改 id 要改名；双链要更新 |
| **两者一致**（推荐新稿） | 追踪最省事 | 标题很长时文件名也长 |

**约定：**

- **真源是 YAML 的 `id`**，脚本、总表、WP 映射都以它为准。  
- 文件名前缀与 `id` 相同，例如 `T01-分子模拟工作平台搭建.md`。  
- `images/articles/.../` 目录仍用 **slug（标题）** 或 `wp_slug`，不必用 `T01` 作文件夹名（避免改编号要搬图）。

#### 三大栏目并列（重要）

站点内容分 **三个同级栏目**，不是 `在线资源/04`、`/05`：

| 仓库目录 | YAML `series` | id 前缀 | WP 一级分类 |
|----------|---------------|---------|-------------|
| `在线资源/`（内分 00–02） | `在线资源` | K / T / C | 在线资源 → 知识/技术/实战案例 |
| `在线工具/` | `在线工具` | **M**（MDStudio） | **在线工具** |
| `解决方案/` | `解决方案` | **S** | **解决方案** |

#### 编号规则（在线资源内仍用 00–02）

| 在线资源子层 | `tier` | id 前缀 |
|-------------|--------|---------|
| 00 知识 | `00-知识文档` | K |
| 01 技术 | `01-技术文档` | T |
| 02 实战 | `02-实战案例` 或 `实战案例` | C（**仅**分子模拟相关案例；工具速成归 T） |

在线工具 / 解决方案：**不设** `00-` 式 tier，用 `series` + `topic` 即可。

#### paywall（YAML）与站点展示名

正文可读性与资源包权益用**同一个** `paywall` 字段表达（四种，互斥）。站点卡片 / 角标用「展示名」；悬停或筛选可用「一句话」。

| YAML（写入 md）     | 站点展示名    | 一句话                     | 适用                                 |
| --------------- | -------- | ----------------------- | ---------------------------------- |
| `free`          | **免费**   | 全文公开；若有小附件也公开链          | 知识/技术引流、无包或小包公开                    |
| `vip`           | **VIP**  | 开通会员后阅读全文               | 知识/技术进阶；**有资源包的非案例文**也只用本档或 `free` |
| `download-vip`  | **VIP下载** | 正文全免费；配套资源包 VIP 免下载        | **资源案例**（代码 / Notebook 包）          |
| `download-paid` | **付费下载** | 正文全免费；资源包单独购买，**会员享折扣** | **资源案例**（大数据 / 轨迹包等）               |

**过渡别名（旧稿可暂留，新稿勿再用）：**

| 旧值 | 视为 |
|------|------|
| `none` | → `free` |
| `vip-partial` | → 改为 `vip`（预览策略见下；勿再写 partial） |
| `resource` | → 按附件策略改为 `download-vip` 或 `download-paid` |

**写作 / erphpdown 约定：**

| paywall | 正文怎么写 |
|---------|------------|
| `free` | 无付费短代码；附件用媒体库绝对 URL |
| `vip` | 导语可留一小段免费钩子；**主体全文**用 `[erphpdown]`（VIP 类型）包裹。勿在 WP 整篇勾「VIP 可见」 |
| `download-vip` | 正文无 VIP 包裹；下载区 `[erphpdown]` 设为 **VIP 免费下载** |
| `download-paid` | 正文无 VIP 包裹；下载区 `[erphpdown]` 设为 **购买下载**，后台为 VIP 配折扣价 |

**栏目默认（建议）：**

| 栏目 | 默认 paywall | 说明 |
|------|--------------|------|
| 00 知识、01 技术 | `free` 为主；少数进阶 `vip` | 有环境包等：公开则仍 `free`，或全文 `vip`（**不要**对技术文单独开 download-*，留给案例） |
| 02 实战 · 资源案例 | `download-vip` 或 `download-paid` | 正文免费引流；用包转化会员或单次付费 |
| 02 实战 · 无重包教程 | `free` 或 `vip` | 与知识/技术相同逻辑 |

**示范：** 平台搭建等公开文 → `free`；会员专享长文 → `vip`；案例代码包 → `download-vip`；大轨迹包 → `download-paid`。

#### 免费附件（如 myenv.zip）

1. 源文件维护在 `MolSimulX-article/downloads/`（`myenv.yml` → `zip -j myenv.zip myenv.yml`）。
2. 上传至 WordPress **媒体库**，建议固定路径：`wp-content/uploads/molsimulx/downloads/myenv.zip`（可在媒体库中按此命名/整理）。
3. T01 正文中用 Markdown 绝对链接：
   - 生产：`https://www.molsimulx.com/wp-content/uploads/molsimulx/downloads/myenv.zip`
   - 本地：`http://molsimulx.local/wp-content/uploads/molsimulx/downloads/myenv.zip`
4. 更新 zip 后**覆盖上传**同路径文件，文中 URL 不必改；再 `publish.py T01` 同步正文即可。

### 4. 付费与 VIP（怎么理解四种 paywall）

读者侧只认四个标签：**免费 / VIP / VIP下载 / 付费下载**（与 YAML 对照见上一节表）。

| 展示名      | 钱从哪来        | 和另外几档的关系                         |
| -------- | ----------- | -------------------------------- |
| **免费**   | 无           | 引流与 SEO                          |
| **VIP**  | 会员订阅        | 买的是**读全文**；非案例文有包时也归这里或免费，不另开下载档 |

### 5. 文章互链（review 期间）

写作时的锚文字、通俗化、禁止正文出现 T##/E## 等约定见 本手册「写作规范」。

- **本地一律用相对路径** `.md`，例如 `[Git简明使用教程](Git简明使用教程.md)`。  
- **不要**在源稿里写 WordPress URL；发布时由 `publish.py` 替换为 `permalink`。  
- **未发布篇目**（YAML 尚无 `wp_post_id`）：线上显示为灰色虚线「待发布」，**不可点击**（避免 404）。  
- **已发布篇目**：本地 YAML 有 `wp_post_id` 时，自动转为 `http://站点/slug/` 可点击链接。  
- **WP 已下架**：请删掉该文 YAML 里的 `wp_post_id`（并标 `archived`）；`publish-cache.json` 会在下次发布时自动剔除过期映射。然后**重发引用它的文章**。  
- 目标文首次发布并写入 `wp_post_id` 后，**需重新发布引用它的文章**，链接才会变可点（不会自动改其他 WP 文章）。**批量首发**时若 A 链到 B，应先发 B（或先发一轮后再发引用方一轮）。  
- review 期间 Obsidian 里仍可链到 `draft` 文章；仅影响线上 HTML。  
- 全局脉络放在本手册「内容规划」，**不要**用公开总索引暴露完整路线图。

#### 批量发布与互链

1. **第一轮**：按依赖顺序发布被引用较多的基础篇（如平台搭建、SSH、Git）。  
2. **第二轮**：发布工作流、示例等互链多的篇目，或对整个 ID 列表再跑一遍 `--write-back-id`（`revised` 会 UPDATE，已 `published` 本地未改 status 的会跳过）。  
3. **`--all-reviewed`** 不会替你排序依赖，只按文件路径遍历；互链敏感时**优先用手写 ID 列表**。

#### 发布顺序建议

`00` → `01` → `02` → `03`，减少站内死链。维护 `slug` 或 `wp_post_id` 映射表（可在内部规划或将来 `publish-cache.json`）。

命名前缀：`hero-` / `fig-` / `screenshot-` / `diagram-`。

#### 懒人工作流

1. 把图放进 `images/` 或 `images/inbox/`。  
2. 在 md 里引用（路径可先随意，只要能被脚本找到）。  
3. 运行整理脚本：

```bash
conda activate molsimulx

# 单篇 + 回写 md 为规范 web 路径
python tools/sync_article_images.py \
  --file 在线资源/01-技术文档/分子模拟工作平台搭建.md \
  --rewrite-md

# 全库预览
python tools/sync_article_images.py --all --dry-run

# 全库整理
python tools/sync_article_images.py --all --rewrite-md
```

4. **md 最终引用** `.../web/*.webp`；**上传 WordPress 时**再把 HTML 里的 `src` 换成媒体库 URL。  
5. **发布默认**在 WebP 生成后删除 `images/` 散落重复源图；**`articles/.../original/` 归档原图保留**。加 `--keep-image-sources` 可连散落源图也不删。

详见 [images/README.md](images/README.md)。全库重转 WebP：`python tools/optimize_images_for_web.py`。

**封面观感：** 发布时正文第一张图会加上 `molsimulx-cover`；封面与正文插图显示高度上限均为 **420px**（比例自由、不裁切）。导语下不要解说封面；需要时在正文结合描述。站点样式由 mu-plugin 全局注入，已上架文章刷新即可生效。

### 7. WordPress 文章模板（Kadence）

母版文章：**ID `650`**（`template-20260707`）— [本地预览](http://molsimulx.local/?p=650)。后期可能更换母版，以第 12 节配置表为准。

结构示意：

```text
行布局（第 1 个）
行布局（第 2 个）
  └─ 章节
       └─ 自定义 HTML    ← Markdown 转 HTML 注入此处
页脚（区块）
```

**发布时不能只 POST 裸 HTML**，应：

1. 读取模板 `650` 的完整区块 `content.raw`（或导出到仓库的 `.blocks.html` 母版文件）。  
2. **仅替换**第二个行布局内「自定义 HTML」块的内容。  
3. CREATE / UPDATE 目标文章，`content` = 整段区块序列。  
4. 模板文 **650 保持 draft/private**，不对外发布。

配置示例（将来发布工具）：

```yaml
template_post_id: 650
# 或 template_file: wordpress/templates/article-650.blocks.html
```

若站点另有 Kadence 文章大标题（hero），可能与模板重复，需对新文设置 `_kad_post_title: hide`（与母版一致）；**标题由 `publish.py` 在正文顶部注入居中 `<h1>`**。

#### Kadence 文章设置（发布标准）

以下与你在 WP 后台右侧 **文章设置** 中调好的选项一致；`publish.py` 每次发布都会写入 `tools/publish.config.yaml` 的 `kadence_meta`：

| 后台选项 | 取值 | post meta |
|----------|------|-----------|
| 透明页眉 | **启用** | `_kad_post_transparent: enable` |
| 文章标题 | **禁用** | `_kad_post_title: hide` |
| 文章布局 | **全宽** | `_kad_post_layout: fullwidth` |
| 内容风格 | **非盒装** | `_kad_post_content_style: unboxed` |
| 内容垂直间距 | **禁用** | `_kad_post_vertical_padding: hide` |
| 显示特色图片 | **禁用** | `_kad_post_feature: hide` |
| 禁用页眉 | **关** | `_kad_post_header: false` |
| 禁用页脚 | **开** | `_kad_post_footer: true`（Kadence：`true` = 禁用页脚） |

另：`strip_template_footer: true` 会去掉母版 650 末尾的 `wp:block` 页脚引用。
