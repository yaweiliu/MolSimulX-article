> **系列标签：** `技术文档` · `工作流` · `归档` · `备份`

一次 NPT 跑下来，轨迹轻松 **几 GB**，而 `in.npt` 和脚本往往只有几 KB。磁盘满了写不进去、scratch 被管理员清空、本地和集群各一份不知道以谁为准——这些比「LAMMPS 报错」更搞心态，而且**重跑一遍可能又是一周**。

本文讲分子模拟数据**放哪、记啥、备啥**：跟 [科研项目目录结构规范](T15-科研项目目录结构规范.md) 的文件夹配套，跟 [Git简明使用教程](T04-Git简明使用教程.md) 分工——**Git 管怎么复现，备份管跑出来的大文件**。

学完能带走：六层数据怎么分（配置、结构、原始轨迹、处理后、图表、元数据各进不进 Git）、本机 / `$HOME` / `$SCRATCH` 各放啥、跑完一轮 runlog 该记哪些字段，以及课题结了怎么冷备份。文件夹长什么样先看 [科研项目目录结构规范](T15-科研项目目录结构规范.md)；本地和集群之间倒文件见 [本地与集群文件传输](T09-本地与集群文件传输.md)。

| 层级 | 进 Git？ | 谁管 |
|------|----------|------|
| `in` 文件、力场、runlog | ✅ | [Git简明使用教程](T04-Git简明使用教程.md) |
| dump、xtc 等原始轨迹 | ❌ | **本篇**（备份 + scratch） |
| 论文用 PDF 图 | ✅ | 跟 `figures/` 一起提交 |

## 一、先分层：什么进 Git，什么别进

把项目里的文件分成六层，一眼知道该怎么管：

| 层级 | 是什么 | 大概多大 | 进 Git？ |
|------|--------|----------|----------|
| **L0 配置** | 力场、`in` 文件、`params.yaml` | KB | ✅ 要 |
| **L1 结构** | `system.data`、xyz、pdb | KB–MB | ✅ 要 |
| **L2 原始输出** | dump、xtc、dcd、`log.lammps` | GB–TB | ❌ 不要 |
| **L3 处理后** | RDF 的 csv、平均密度表 | MB 级 | ⚠️ 小的可进，大的别进 |
| **L4 图表** | 论文用 PDF、PNG | 通常 MB 内 | ✅ 要 |
| **L5 元数据** | runlog、JOBID、随机种子、路径说明 | KB | ✅ 要 |

**记一句：** Git 回答「**怎么再跑一遍**」；L2 轨迹回答「**那次跑出来的数**」——两件事都要，但不能全塞进 Git。

## 二、放哪儿：本机、集群、归档

### 1. 本地笔记本 / 工作站

```
项目/data/raw/          # 正在分析的轨迹（.gitignore 掉）
项目/data/processed/    # 从轨迹抽出的小表
外置硬盘 / 组 NAS / 网盘   # 跑完项目的「冷备份」
```

SSD 就几百 GB：**在做的项目**留本地；课题结了，整包 `raw` 挪到移动硬盘或组里存储，给别的项目腾地方。

### 2. 计算集群：家目录 vs scratch

各校名字不一样，但逻辑类似：

| 位置 | 常见环境变量 / 路径 | 特点 | 建议放啥 |
|------|---------------------|------|----------|
| **家目录** | `$HOME` | 配额小、一般不随便清 | 代码、`lammps/`、脚本、小结构 |
| **高速临时盘** | `$SCRATCH`、`$WORK`、`/scratch/用户名` | 空间大、**可能定期清理** | 跑模拟时的 dump、大轨迹 |
| **组共享目录** | 问管理员 | 组内规范各异 | 按课题组规定 |

> **Tips：** **以你所在集群的用户手册为准**——`echo $SCRATCH`、`df -h`、`quota -s` 先跑一遍，搞清家目录多大、scratch 多久清一次、清之前会不会发邮件。别凭别的学校的经验硬套路径。

**交作业前**确认 `#SBATCH` 或 `in` 里轨迹写在 scratch（或手册指定的盘），别默认写 `$HOME`，跑一半「磁盘满」或把家目录塞爆。

### 3. 本地 ↔ 集群怎么倒

上传输入、拉回结果见 [本地与集群文件传输](T09-本地与集群文件传输.md)。常用套路：

```bash
# 上传项目（别把本地 data/raw 又传回去）
rsync -avh --progress --exclude 'data/raw/' ./ molcluster:~/project/

# 拉回已处理的小文件
rsync -avh --progress molcluster:~/project/data/processed/ ./data/processed/

# 拉回某次 run 的 log（轨迹太大可只拉 log + 最后几帧）
scp molcluster:~/project/data/raw/run030/log.lammps ./data/raw/run030/
```

作业怎么交、输出写哪，见 [集群与SLURM简明教程](T10-集群与SLURM简明教程.md)。

## 三、元数据：轨迹在，解释也要在

光有 `dump.lammpstrj`，半年后你可能忘了：用的哪个力场、种子多少、对应论文哪张图。在 **`docs/runlog.md`**（模板见 [科研项目目录结构规范](T15-科研项目目录结构规范.md)）里，**每次重要运行**记一笔：

```markdown
## 2026-06-30 NPT 300K — run030

- **JOBID：** 12345（集群 molcluster）
- **输入：** `lammps/in.npt`，`structures/system.data`
- **力场：** TIP3P
- **系综：** NPT，300 K，1 atm，2 fs，1e6 steps
- **随机种子：** 12345
- **输出路径：** `/scratch/user/proj/run030/`（集群上的真实路径）
- **Git commit：** `a1b2c3d`
- **状态：** 完成；密度 0.99 g/cm³
- **下游分析：** `notebooks/02_rdf.ipynb` → `figures/fig1_rdf.pdf`
```

也可用 [Obsidian知识库搭建](T13-Obsidian知识库搭建.md) 记笔记，但**路径、JOBID、commit** 这几项别省。审稿人要补充说明、你要重画一张图，都靠这张「账单」。

## 四、备份：3-2-1 的科研简化版

完整 **3-2-1**（三份、两种介质、一份异地）理想很满，日常可以这么落地：

| 思路 | 分子模拟里怎么做 |
|------|------------------|
| **至少两份** | 集群 scratch 一份 + 模拟结束后 `rsync` 到本地或组 NAS |
| **两种介质** | 笔记本磁盘 + 外置硬盘 / 网盘 |
| **一份「离机房远点」** | 网盘、家里硬盘、学校备份服务（看政策） |

**优先保什么（按顺序）：**

1. **L0–L1 + 代码** —— 推送到 Git 远程（GitHub / Gitee）就是一份备份  
2. **L5 元数据** —— 跟 Git 一起走  
3. **L2 轨迹** —— 跑**结束尽快**归档；不必永远留每一次失败试跑的全轨迹  

**可以删的：** 已记在 runlog 里「为何失败」的调试 dump、重复试跑中间帧——但留一份 `log.lammps` 方便以后写 Methods。

### 常用命令

```bash
# 增量同步到外置盘
rsync -avh --progress ~/project/data/raw/ /Volumes/Backup/md-project-raw/

# 单次 run 打 tar 包（便于挪位置）
tar czvf run030_raw.tar.gz -C data/raw run030/

# 看谁占空间
du -sh ~/project/*/ | sort -h
```

网盘、OneDrive、百度网盘、课题组 NAS——按学校规定选；**未发表数据**设密码、别公开分享链接。

发表后常把代表性轨迹/输入上传到 **Zenodo、Figshare** 等拿 DOI，README 里写链接（期刊补充材料也会要）。

## 五、Git 与 Git LFS：别拿 Git 当网盘

- **普通 Git**：跟踪 L0、L1、L4、L5、脚本、Notebook（清输出，见 [Jupyter Notebook科研使用规范](T16-Jupyter Notebook科研使用规范.md)）  
- **Git LFS**：几十 MB 级的结构或中等轨迹**若组内强制版本化**时可用  

```bash
git lfs install
git lfs track "*.xtc"
git add .gitattributes
```

**几 GB 的 dump 仍然不要进 Git**——用 runlog 写清路径 + 外部备份。`git push` 卡住、仓库克隆要一小时，基本都是误提交了大数据。

## 六、一条数据的一生（从计划到发表）

```
写 in / 结构 → git commit
    → sbatch，轨迹落 scratch
    → runlog 记 JOBID + 路径
    → 分析：processed + figures → 再 commit
    → 模拟结束：rsync raw 到本地 / NAS（释 scratch 配额）
    → 投稿：补充材料 + 图在 figures/
    → 发表：Zenodo 等归档 + README 链 DOI
```

| 阶段 | 你该干啥 |
|------|----------|
| **跑着** | scratch 上写输出；runlog 随跑随更 |
| **分析完** | 图进 `figures/`；csv 进 `processed/` |
| **课题段落** | tar 打包 raw 移出 scratch；删明确失败的垃圾 run |
| **要投稿** | 核对 runlog 与论文图、表一致 |
| **已发表** | 按期刊要求上传数据；仓库 README 更新 DOI |

## 七、安全与合规（别踩雷）

- 未发表结果：**私有 Git** 或本地；公开网盘别不设权限  
- 合作课题：说清楚 scratch 谁有权删、备份放哪  
- 公开 README 里别写学号、手机号、集群内部绝对路径（写相对路径 + runlog 里记全）  
- 涉密或合同项目：按单位要求，可能禁止公有云备份

## 八、常见问题

### 1. scratch 被清空了怎么办？

- **输入 + 参数**在 Git 和 runlog 里 → 可以**重跑**（要花机时）  
- **从没备份的轨迹** → 基本找不回来  

教训：**作业一正常结束就 rsync 关键 run**，别等「毕业前再整理」。

### 2. 磁盘 / 配额满了

```bash
quota -s              # 集群配额（若有）
du -sh data/raw/* | sort -h
```

先删：失败试跑、重复 dump、已归档过的旧 run；保留：当前在写的、论文直接引用的。

### 3. 本地和集群两份，以谁为准？

以 **Git 最新 commit + runlog 里的 JOBID** 为准。分析前对一下：

```bash
git log -1 --oneline
# 和 docs/runlog.md 里这次 run 的 commit 是否一致
```

### 4. 轨迹要不要压缩？

`tar.gz` 省空间、适合冷备份；**日常分析**用未压缩或集群上原格式，免得每次分析都解压半天。

### 5. 只备份 log 不备份轨迹行吗？

写论文只要一张 RDF 图——可以只长期留 **processed + figures**；但要防审稿人要看完整轨迹，或你要换分析方法。**关键体系建议至少留一份 raw 冷备份**。

## 九、小结

1. **分层**：代码进 Git，GB 级轨迹进 scratch / 外置盘，别混。  
2. **runlog** 记全 JOBID、路径、种子、commit、对应哪张图。  
3. **模拟结束就备份** raw；集群清 scratch 前务必确认已 `rsync`。  
4. **集群目录名各校不同**——以手册为准，`quota` / `du` 常看。  
5. 与 [本地与集群文件传输](T09-本地与集群文件传输.md)、[科研项目目录结构规范](T15-科研项目目录结构规范.md)、[从模拟到论文图的工作流](T18-从模拟到论文图的工作流.md) 一起用。
