## 快速发布指南

> 只记命令时用本节；状态、YAML、付费、模板等细节见下文各章。

## 一次性准备

```bash
cd /path/to/MolSimulX网站

conda activate molsimulx
# 若尚未创建：conda create -n molsimulx python=3.12 -y && pip install -r tools/requirements-publish.txt
pip install -r tools/requirements-publish.txt

cp .env.local.example .env.local
# 编辑 .env.local：WP_SITE_URL、WP_USERNAME、WP_APP_PASSWORD、WP_TEMPLATE_POST_ID=650
```

## 日常流程（单篇）

```bash
# 1. 看编号与 status
python tools/publish.py --list

# 2. 审阅通过后，在 md 文首 YAML 把 status 改为 reviewed（或 revised）

# 3. 预览转换（不上传 WP、不整理图）
python tools/publish.py T01 --dry-run

# 4. 发布到 WP 草稿（自动整理配图 + 注入模板 650；默认 WP 状态 draft）
python tools/publish.py T01 --write-back-id

# 5. 到 WP 后台预览 → 配 erphpdown → 人工点「发布」
# 6. 本地 YAML 改 status: published，填写 wp_post_id、published_at
```

`T01` 可换成任意编号；也可用 `--id T01`、`--pick`（交互选篇）、`--file Git简明使用教程.md`（唯一文件名）。

## 日常流程（批量）

`publish.py` **已支持一次处理多篇**：按顺序逐篇执行配图同步 → 转 HTML → 上传 WP；结束时打印 **批量发布汇总**（成功 / 跳过 / 失败）。

```bash
# 指定多篇（空格分隔）
python tools/publish.py T02 T03 T04 --write-back-id

# 或用 --id（支持逗号、空格：T02,T03 或 T02 T03）
python tools/publish.py --id T02,T03,T04 --write-back-id

# 全库扫描：遍历 在线资源 / 在线工具 / 解决方案 下全部教程 md
# 仅 status=reviewed / revised 的会真正同步，其余在汇总里显示「跳过」
python tools/publish.py --all-reviewed --write-back-id

# 批量预览（不上传 WP）
python tools/publish.py T01 T02 T03 --dry-run
```

**批量时注意：**

| 项 | 说明 |
|----|------|
| **status 门禁** | 每篇独立判断；`draft` / `published` 默认跳过，汇总里可见原因 |
| **共用参数** | `--dry-run`、`--write-back-id`、`--publish`、`--force`、`--skip-sync-images` 等对**整批**生效 |
| **退出码** | 任一篇 HTTP/校验失败则进程退出码为 `1`；成功的篇目不受影响 |
| **发布顺序** | 显式指定 ID 时按你写的顺序；`--all-reviewed` 按目录树路径排序 |
| **站内互链** | 被引用文尚无 `wp_post_id` 时，引用方线上仍为「待发布」；批量首发后需**再发一遍引用它的文章**（见第 5 节） |

建议批量首发：`00` 知识 → `01` 技术 → `02` 示例`，或按 [内容规划](内容规划.md) 学习路径指定 ID 列表，减少站内灰链。

## 配图（可单独跑；`publish.py` 默认会自动执行）

```bash
# 单篇：归档 inbox → 转 WebP → 回写 md 路径
python tools/sync_article_images.py T01 --rewrite-md

# 与发布一致：清理 images/ 散落重复源图（original/ 保留）
python tools/sync_article_images.py T01 --rewrite-md --prune-sources

# 全库预览 / 整理
python tools/sync_article_images.py --all --dry-run
python tools/sync_article_images.py --all --rewrite-md
```

**发布默认清散落图：** `publish.py` 在 WebP 成功后删除 `images/` 根目录等重复件；**`articles/.../original/` 归档原图保留**。连散落源图也保留时加 `--keep-image-sources`。

## 状态与发布门禁

| 本地 `status` | `publish.py` 是否同步 |
|---------------|----------------------|
| `draft` | 否（除非 `--force`） |
| `reviewed` | 是 → WP **CREATE** draft |
| `revised` | 是 → WP **UPDATE** draft（需已有 `wp_post_id`） |
| `published` | 否 |

## 母版 650（版式 / 尺寸）

- **在线发布**（`publish.py T01`）：每次从 WP **实时读取**母版 650，再注入正文；你在后台改的宽度、间距会生效。
- **更新已有文章**：默认**整篇重新套用**当前母版（不是只改正文）。若要在 WP 里手工改过版式并保留，加 `--keep-layout`。
- **dry-run 预览**：用 `tools/.publish-template-cache.html`；要与线上一致时先发布一次，或 `python tools/publish.py T01 --dry-run --refresh-template`。

`publish-cache.json` **不再缓存母版**（避免用到旧版式）。

## 出问题时

```bash
# 只看配图报告
python tools/sync_article_images.py T01 --dry-run

# 强制 draft 状态写入 WP（覆盖 .env 默认）
python tools/publish.py T01 --status draft

# 批量中某篇失败：看终端末尾「批量发布汇总」，修正后单独重跑该 ID
python tools/publish.py T08 --write-back-id
```

## 1. 内容仓库分工

| 位置 | 用途 | 是否上架 |
|------|------|----------|
| `在线资源/` | 教程 Markdown 源稿（Obsidian） | 发布后内容进 WP，源稿留本地 |
| `images/` | 配图（可先随意放，脚本归档） | 上传媒体库，md 仍用本地相对路径 |
| `downloads/`、`codes/` | 环境 yml、zip 等资源包源文件 | 上传 WP 媒体库，文中用**绝对 URL** 链出（免费资源不再用 erphpdown） |
| `写作规范.md` / `发布与上线.md` / `内容规划.md` | 写作、发布、选题规划 | **不上架** |
| `在线资源/资源导航.md` | 站点短入口文案 | 可上架为固定页 |
| Local WP 站点（`~/Local Sites/molsimulx` 等） | WordPress 运行环境 | 线上/本地；**不进本仓库** |

**原则：** Markdown 是正文唯一源；WordPress 只接收发布产物（区块 + HTML + 媒体），不把整站 WP 拷进 `在线资源/`。

## 2. 文章状态（工作流）

| status | 含义 | 对外 | 发布动作 |
|--------|------|------|----------|
| `draft` | AI 初稿 / 审阅中 | 否 | 不发布 |
| `reviewed` | 首次人工审阅通过 | 否 | **CREATE**（新建 WP 文章） |
| `published` | 已上线，本地与线上一致 | 是 | 无 |
| `revised` | 已上线但本地又修改 | 线上仍为旧版 | **UPDATE**（同 `wp_post_id`） |
| `archived` | 下线或废弃 | 否 | 不发布 / 可选下线 |

### 状态流转

```text
draft ──审完──► reviewed ──首次发布──► published
  ▲                                      │
  └──────── 大改退回 draft                │ 本地修改
                                          ▼
                                    revised ──再审 + 更新发布──► published
```

- **`reviewed`**：只用于第一次上线。  
- **`revised`**：只用于已发布后的修改；更新成功后改回 `published`。  
- 自动发布工具（若实现）应：**`reviewed` / `revised` → 写入 WP 均为 `draft`**；人工在后台点「发布」后，本地才改 `published`。

### WordPress 侧：先草稿，再人工发布

```text
本地 status=reviewed（或 revised）
        ↓
脚本 / 手工同步 → WP 文章状态 = draft（草稿）
        ↓
WP 后台预览（模板 650、图片、erphpdown、VIP）
        ↓
人工点击「发布」
        ↓
本地 status=published，填写 wp_post_id、published_at
```

| 阶段 | 本地 YAML | WordPress |
|------|-----------|-----------|
| 写作审阅 | `draft` | （无或旧版） |
| 可上线 | `reviewed` | 待创建 **draft** |
| 已同步待确认 | `reviewed` | **draft**（预览） |
| 正式上线 | `published` | **publish** |
| 改稿 | `revised` | 再次 **draft** 或预览后更新（见下） |

**修订说明：** 已上线的文章若在 WP 里改回 `draft` 会暂时下架。可选做法：（1）严格审核：修订也先 draft，确认后再发布；（2）已熟内容：用 WP「预览修订」插件或仅在本地 `revised` 审完后再点一次「更新」。默认约定：**自动工具只创建/更新 draft，不直接 publish**。

## 3. 文章编号、文件名与文首 YAML

### 文件名要不要加 `T01-` 前缀？

**建议：新文章可以加，旧文不必一次性重命名。**

| 做法 | 优点 | 缺点 |
|------|------|------|
| **仅 YAML 里写 `id: T01`** | 文件名仍可读；Obsidian 双链不用改 | 资源管理器里不按编号排序 |
| **文件名 `T01-标题.md`** | 文件夹里一眼排序、好追踪 | 改 id 要改名；双链要更新 |
| **两者一致**（推荐新稿） | 追踪最省事 | 标题很长时文件名也长 |

**约定：**

- **真源是 YAML 的 `id`**，脚本、总表、WP 映射都以它为准。  
- 文件名前缀与 `id` 相同，例如 `T01-分子模拟工作平台搭建.md`。  
- `images/articles/.../` 目录与 **md 文件名**一致（含 `M07` / `T01` 等编号），便于查找；**不用** `wp_slug` 作文件夹名。

### 三大栏目并列（重要）

站点内容分 **三个同级栏目**，不是 `在线资源/04`、`/05`：

| 仓库目录 | YAML `series` | id 前缀 | WP 一级分类 |
|----------|---------------|---------|-------------|
| `在线资源/`（内分 00–02） | `在线资源` | K / T / C | 在线资源 → 知识/技术/实战案例 |
| `在线工具/`（内分 MDStudio / DataHub / AILab） | `在线工具` | **M**（等） | **在线工具** → MDStudio / DataHub / AILab |
| `解决方案/` | `解决方案` | **S** | **解决方案** |

### 编号规则（在线资源内仍用 00–02）

| 在线资源子层 | `tier` | id 前缀 |
|-------------|--------|---------|
| 00 知识 | `00-知识文档` | K |
| 01 技术 | `01-技术文档` | T |
| 02 实战 | `02-实战案例` 或 `实战案例` | C（**仅**分子模拟相关案例；工具速成归 T） |

### 在线工具子产品（与 WP 二级分类同名）

| 仓库子目录 | YAML `tier` | WP 二级 | 说明 |
|------------|-------------|---------|------|
| `00-MDStudio/` | `MDStudio` | MDStudio | 浏览器装 Lammps 输入等 |
| `01-DataHub/` | `DataHub` | DataHub | 预留 |
| `02-AILab/` | `AILab` | AILab | 预留 |

`topic` 可与 `tier` 同写产品名；系列标签建议以产品名打头（如 `` `MDStudio` · `Quickstart` · … ``）。

| 栏目 | 文件名示例 |
|------|------------|
| 在线资源 | `T01-分子模拟工作平台搭建.md` |
| 在线工具 | `00-MDStudio/M01-Quickstart….md` |
| 解决方案 | `S01-从PDB到Lammps输入.md` |

### 发布后填写（首发成功后再写）

```yaml

```

修订时再加：`revised_at`、`revision_note`；改回 `published` 后清空 `revision_note`。

### paywall（YAML）与站点展示名

正文可读性与资源包权益用**同一个** `paywall` 字段表达（四种，互斥）。站点卡片 / 角标用「展示名」；悬停或筛选可用「一句话」。

| YAML（写入 md）     | 站点展示名    | 一句话                     | 适用                                 |
| --------------- | -------- | ----------------------- | ---------------------------------- |
| `free`          | **免费**   | 全文公开；若有小附件也公开链          | 知识/技术引流、无包或小包公开                    |
| `vip`           | **VIP**  | 开通会员后阅读全文               | 知识/技术进阶；**有资源包的非案例文**也只用本档或 `free` |
| `download-vip`  | **VIP下载** | 正文全免费；配套资源包 VIP 免下载        | **资源案例**（代码 / Notebook 包）          |
| `download-paid` | **付费下载** | 正文全免费；资源包单独购买，**会员享折扣** | **资源案例**（大数据 / 轨迹包等）               |

**过渡别名（旧稿可暂留，新稿勿再用）：**

| 旧值 | 视为 |
|------|------|
| `none` | → `free` |
| `vip-partial` | → 改为 `vip`（预览策略见下；勿再写 partial） |
| `resource` | → 按附件策略改为 `download-vip` 或 `download-paid` |

**写作 / erphpdown 约定：**

| paywall | 正文怎么写 |
|---------|------------|
| `free` | 无付费短代码；附件用媒体库绝对 URL |
| `vip` | 导语可留一小段免费钩子；**主体全文**用 `[erphpdown]`（VIP 类型）包裹。勿在 WP 整篇勾「VIP 可见」 |
| `download-vip` | 正文无 VIP 包裹；下载区 `[erphpdown]` 设为 **VIP 免费下载** |
| `download-paid` | 正文无 VIP 包裹；下载区 `[erphpdown]` 设为 **购买下载**，后台为 VIP 配折扣价 |

**栏目默认（建议）：**

| 栏目 | 默认 paywall | 说明 |
|------|--------------|------|
| 00 知识、01 技术 | `free` 为主；少数进阶 `vip` | 有环境包等：公开则仍 `free`，或全文 `vip`（**不要**对技术文单独开 download-*，留给案例） |
| 02 实战 · 资源案例 | `download-vip` 或 `download-paid` | 正文免费引流；用包转化会员或单次付费 |
| 02 实战 · 无重包教程 | `free` 或 `vip` | 与知识/技术相同逻辑 |

**示范：** 平台搭建等公开文 → `free`；会员专享长文 → `vip`；案例代码包 → `download-vip`；大轨迹包 → `download-paid`。

### 免费附件（如 myenv.zip）

1. 源文件维护在 `MolSimulX-article/downloads/`（`myenv.yml` → `zip -j myenv.zip myenv.yml`）。
2. 上传至 WordPress **媒体库**，建议固定路径：`wp-content/uploads/molsimulx/downloads/myenv.zip`（可在媒体库中按此命名/整理）。
3. T01 正文中用 Markdown 绝对链接：
   - 生产：`https://www.molsimulx.com/wp-content/uploads/molsimulx/downloads/myenv.zip`
   - 本地：`http://molsimulx.local/wp-content/uploads/molsimulx/downloads/myenv.zip`
4. 更新 zip 后**覆盖上传**同路径文件，文中 URL 不必改；再 `publish.py T01` 同步正文即可。

## 4. 付费与 VIP（怎么理解四种 paywall）

读者侧只认四个标签：**免费 / VIP / VIP下载 / 付费下载**（与 YAML 对照见上一节表）。

| 展示名      | 钱从哪来        | 和另外几档的关系                         |
| -------- | ----------- | -------------------------------- |
| **免费**   | 无           | 引流与 SEO                          |
| **VIP**  | 会员订阅        | 买的是**读全文**；非案例文有包时也归这里或免费，不另开下载档 |
| **VIP下载** | 含在 VIP 里（免下载码） | 正文故意免费，转化点在**包**                 |
| **付费下载** | 单次购包；会员折扣   | 正文免费；大资源不进月费白送                   |

**注意：** 勿在 WordPress 把整篇文章勾选为「VIP 可见」；仅 `erphpdown` 块内需要门禁。

## VIP 全文结构（`paywall: vip`）

```markdown
（导语数段，可免费——须按「写作规范 · 1.1」写厚：场景钩子 + 能带走什么；VIP 篇墙外几乎是整篇门面）

[erphpdown]

## 一、……
（全文主体至小结前）

[/erphpdown]

## 学习路径
（建议仍免费，便于站内互链）
```

## 资源案例结构（`download-vip` / `download-paid`）

```markdown
（全文免费：步骤、图、说明）

## 资源下载
[erphpdown]
（download-vip：VIP 免费下载；download-paid：购买下载 + VIP 折扣）
[/erphpdown]

## 学习路径
```

旧稿若仍写「自二、起部分 VIP」，发布前改为上表四种之一并改短代码范围。

## 5. 文章互链（review 期间）

写作时的锚文字、通俗化、禁止正文出现 T##/E## 等约定见 [写作规范](写作规范.md)。

- **本地一律用相对路径** `.md`，例如 `[Git简明使用教程](Git简明使用教程.md)`。  
- **不要**在源稿里写 WordPress URL；发布时由 `publish.py` 替换为 `permalink`。  
- **未发布篇目**（YAML 尚无 `wp_post_id`）：线上显示为灰色虚线「待发布」，**不可点击**（避免 404）。  
- **已发布篇目**：本地 YAML 有 `wp_post_id` 时，自动转为 `http://站点/slug/` 可点击链接。  
- **WP 已下架**：请删掉该文 YAML 里的 `wp_post_id`（并标 `archived`）；`publish-cache.json` 会在下次发布时自动剔除过期映射。然后**重发引用它的文章**。  
- 目标文首次发布并写入 `wp_post_id` 后，**需重新发布引用它的文章**，链接才会变可点（不会自动改其他 WP 文章）。**批量首发**时若 A 链到 B，应先发 B（或先发一轮后再发引用方一轮）。  
- review 期间 Obsidian 里仍可链到 `draft` 文章；仅影响线上 HTML。  
- 全局脉络放在 [内容规划](内容规划.md)，**不要**用公开总索引暴露完整路线图。

### 批量发布与互链

1. **第一轮**：按依赖顺序发布被引用较多的基础篇（如平台搭建、SSH、Git）。  
2. **第二轮**：发布工作流、示例等互链多的篇目，或对整个 ID 列表再跑一遍 `--write-back-id`（`revised` 会 UPDATE，已 `published` 本地未改 status 的会跳过）。  
3. **`--all-reviewed`** 不会替你排序依赖，只按文件路径遍历；互链敏感时**优先用手写 ID 列表**。

### 发布顺序建议

`00` → `01` → `02` → `03`，减少站内死链。维护 `slug` 或 `wp_post_id` 映射表（可在内部规划或将来 `publish-cache.json`）。

## 6. 配图流程（不必手建目录）

### 目录约定（归档后）

```text
images/
├── inbox/              # 临时丢图（脚本会 move 到对应文章）
├── site/               # 首页、品牌等固定页
└── articles/
    └── {层级}/{md文件名}/   # 例：在线工具/M07-MDStudio周期分子
        ├── original/   # 源图 png/jpg…
        └── web/        # WebP（脚本生成）
```

目录名与 md 文件名一致（含文章编号）；命名前缀：`hero-` / `fig-` / `screenshot-` / `diagram-`。

### 懒人工作流

1. 把图放进 `images/` 或 `images/inbox/`。  
2. 在 md 里引用（路径可先随意，只要能被脚本找到）。  
3. 运行整理脚本：

```bash
conda activate molsimulx

# 单篇 + 回写 md 为规范 web 路径
python tools/sync_article_images.py \
  --file 在线资源/01-技术文档/分子模拟工作平台搭建.md \
  --rewrite-md

# 全库预览
python tools/sync_article_images.py --all --dry-run

# 全库整理
python tools/sync_article_images.py --all --rewrite-md
```

4. **md 最终引用** `.../web/*.webp`；**上传 WordPress 时**再把 HTML 里的 `src` 换成媒体库 URL。  
5. **发布默认**在 WebP 生成后删除 `images/` 散落重复源图；**`articles/.../original/` 归档原图保留**。加 `--keep-image-sources` 可连散落源图也不删。

详见 [images/README.md](images/README.md)。全库重转 WebP：`python tools/optimize_images_for_web.py`。

**封面观感：** 发布时正文第一张图会加上 `molsimulx-cover`；封面与正文插图显示高度上限均为 **420px**（比例自由、不裁切）。导语下不要解说封面；需要时在正文结合描述。站点样式由 mu-plugin 全局注入，已上架文章刷新即可生效。

### 与 FileBird（WP 媒体文件夹）

上传时建议虚拟文件夹与本地对齐，例如：

```text
在线资源 / 01-技术文档 / 分子模拟工作平台搭建
```

FileBird 移动文件**不破坏**已发布 URL；修订时尽量只 UPDATE 文章、复用已有 `attachment_id`（hash 未变则跳过上传）。

## 7. WordPress 文章模板（Kadence）

母版文章：**ID `650`**（`template-20260707`）— [本地预览](http://molsimulx.local/?p=650)。后期可能更换母版，以第 12 节配置表为准。

结构示意：

```text
行布局（第 1 个）
行布局（第 2 个）
  └─ 章节
       └─ 自定义 HTML    ← Markdown 转 HTML 注入此处
页脚（区块）
```

**发布时不能只 POST 裸 HTML**，应：

1. 读取模板 `650` 的完整区块 `content.raw`（或导出到仓库的 `.blocks.html` 母版文件）。  
2. **仅替换**第二个行布局内「自定义 HTML」块的内容。  
3. CREATE / UPDATE 目标文章，`content` = 整段区块序列。  
4. 模板文 **650 保持 draft/private**，不对外发布。

配置示例（将来发布工具）：

```yaml
template_post_id: 650
# 或 template_file: wordpress/templates/article-650.blocks.html
```

若站点另有 Kadence 文章大标题（hero），可能与模板重复，需对新文设置 `_kad_post_title: hide`（与母版一致）；**标题由 `publish.py` 在正文顶部注入居中 `<h1>`**。

### Kadence 文章设置（发布标准）

以下与你在 WP 后台右侧 **文章设置** 中调好的选项一致；`publish.py` 每次发布都会写入 `tools/publish.config.yaml` 的 `kadence_meta`：

| 后台选项 | 取值 | post meta |
|----------|------|-----------|
| 透明页眉 | **启用** | `_kad_post_transparent: enable` |
| 文章标题 | **禁用** | `_kad_post_title: hide` |
| 文章布局 | **全宽** | `_kad_post_layout: fullwidth` |
| 内容风格 | **非盒装** | `_kad_post_content_style: unboxed` |
| 内容垂直间距 | **禁用** | `_kad_post_vertical_padding: hide` |
| 显示特色图片 | **禁用** | `_kad_post_feature: hide` |
| 禁用页眉 | **关** | `_kad_post_header: false` |
| 禁用页脚 | **开** | `_kad_post_footer: true`（Kadence：`true` = 禁用页脚） |

另：`strip_template_footer: true` 会去掉母版 650 末尾的 `wp:block` 页脚引用。

## 8. 发布前检查清单

### 首次发布（`status: reviewed`）

- [ ] YAML：`id`、`tier`、`topic`、`paywall` 已填  
- [ ] 本地 `status` 为 `reviewed`  
- [ ] `sync_article_images.py --rewrite-md` 已跑通  
- [ ] 同步到 WP，状态为 **draft**（不是 publish）  
- [ ] WP 后台预览：模板 650、图片、erphpdown  
- [ ] 人工点击 **发布**  
- [ ] 本地改 `status: published`，填 `wp_post_id`、`published_at`  

### 修订再发（`status: revised`）

- [ ] 必须有 `wp_post_id`  
- [ ] **UPDATE** 同一篇文章；自动工具默认仍为 **draft**，人工确认后再发布  
- [ ] 核对 `erphpdown` 块数量与 `erphpdown_blocks` 一致  
- [ ] 成功后 `status: published`，更新 `published_at` / 清空 `revision_note`  

### 批量同步（多篇 / `--all-reviewed`）

- [ ] 先用 `python tools/publish.py --list` 确认各篇 `status` 为 `reviewed` 或 `revised`  
- [ ] 建议先 `T01 T02 … --dry-run` 看转换与配图报告  
- [ ] 正式同步加 `--write-back-id`；默认仍为 WP **draft**，勿误加 `--publish` 除非确定直接上线  
- [ ] 检查终端末尾 **批量发布汇总**：失败篇单独重跑；跳过篇确认是否故意为 `draft`  
- [ ] WP 后台逐篇或抽样预览 erphpdown、图片、母版 650  
- [ ] 人工发布后，本地各篇改 `status: published` 并填 `published_at`（脚本**不会**批量改 status）  
- [ ] 若存在站内互链，按第 5 节考虑**第二轮**发布引用方文章

## 9. 修订与 erphpdown（订阅会不会丢）

- **VIP 资格、已购订单** 存在 WP/erphpdown 数据库，**不会因 UPDATE 文章而消失**。  
- 风险在于：正文里 **删掉或写坏** `[erphpdown]` 短代码，或 **删文新建** 导致 post ID 变化。  
- 对策：同 ID 更新、发布前校验 `erphpdown_blocks`、付费附件换链后实测下载。

## 10. 工具一览（本项目）

| 工具                                 | 作用                                     | 状态                   |
| ---------------------------------- | -------------------------------------- | -------------------- |
| `tools/sync_article_images.py`     | 按 md 引用归档图片 + 转 WebP + 可选回写路径          | 可用                   |
| `tools/optimize_images_for_web.py` | 扫描 `original/` 批量转 `web/*.webp`        | 可用                   |
| `tools/publish.py` | md → HTML、母版 650 注入、WP REST、状态门禁、**单篇/批量** draft 同步 | **可用**（需 `.env.local`） |

### `publish.py` 快速上手

```bash
conda activate molsimulx
pip install -r tools/requirements-publish.txt

cp .env.local.example .env.local   # 填写 WP_USERNAME、WP_APP_PASSWORD

# 预览（不用打中文路径）
python tools/publish.py --list
python tools/publish.py --id T01 --dry-run
python tools/publish.py T01 --dry-run              # positional 等同 --id

# 单篇同步到 WP 草稿（status 须为 reviewed 或 revised；自动 sync 配图）
python tools/publish.py T01 --write-back-id

# 批量：多篇 positional 或逗号分隔 --id
python tools/publish.py T02 T03 T04 --write-back-id
python tools/publish.py --id T02,T03 --dry-run

# 全库可发布篇（draft/published 等会在汇总中跳过）
python tools/publish.py --all-reviewed --write-back-id

python tools/publish.py --pick                     # 交互选编号（单篇）
```

配置说明：

| 文件 | 用途 |
|------|------|
| `.env.local` | 站点 URL、账号、母版 ID（不进 git） |
| `tools/publish.config.yaml` | 分类映射、剥除行、是否用 Kadence 母版 |
| `tools/templates/article-650.blocks.html` | 母版离线备份（dry-run / 无网时） |
| `tools/publish-cache.json` | 媒体与 slug 缓存（自动生成） |
